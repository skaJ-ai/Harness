# VALUE_REDEF: HARP 제품 가치 재정의

> **문서 목적**: 설계/구현 LLM에게 전달하는 제품 가치 정의서.
> 모든 기능 설계·우선순위 판단·UX 의사결정의 기준점으로 사용한다.
>
> **작성일**: 2026-03-29
> **상태**: Active

---

## 1. 한 줄 정의

> **HARP는 HR 도메인 지식과 보고서 방법론이 내장된, AI 기반 보고서 공방(工房)이다.**
>
> 하네스 엔지니어링 기반으로 품질을 상향 평준화하고,
> 사용할수록 DB가 쌓여 더 나은 결과물을 만들 수 있는 작업 환경.

---

## 2. 이 제품이 존재하는 이유: 이중 간극(Double Gap)

```
[사외 LLM]  ──── 보안 장벽 ────  [사내 LLM]  ──── 숙련도 장벽 ────  [HR 담당자]
  (고성능)       데이터 유출 불가    (저성능)       프롬프팅/컨텍스트     (비전문가)
                                        ↑                구성 불가
                                   HARP가 양쪽 간극을 모두 메운다
```

### 간극 1: 사외 LLM ↔ 사내 LLM (성능 간극)

- 사내 문서·인사 데이터는 사외로 전송할 수 없다.
- 사내 LLM은 성능이 낮다.
- **HARP의 해법**: 컨텍스트 설계로 성능 차이를 보정한다. 잘 구조화된 컨텍스트는 모델 성능 차이를 줄인다.

### 간극 2: LLM ↔ HR 담당자 (숙련도 간극)

- AI 성능이 올라가도, 프롬프팅·컨텍스트 구성을 못 하면 체감 불가.
- 대부분의 HR 담당자는 AI 초보자이다.
- **HARP의 해법**: 도메인 특화 방법론을 하네스에 내장하여 자동 적용한다. 사용자는 "교육 결과 보고서 써줘"만 말하면 된다.

---

## 3. 핵심 가치 삼각형 (Value Triangle)

```
            ┌─────────────────┐
            │  컨텍스트 설계   │
            │ (Context Design) │
            └────────┬────────┘
                     │
        ┌────────────┴────────────┐
        ▼                         ▼
┌───────────────┐       ┌───────────────────┐
│ 방법론 자동적용 │       │  점진적 자산화     │
│ (Auto Method)  │       │ (Progressive Asset)│
└───────────────┘       └───────────────────┘
```

### 3.1 컨텍스트 설계 (Context Design) — 1순위

> **LLM 성능을 믿지 않는다. 컨텍스트를 설계한다.**

- 사내 LLM의 한계를 "입력의 질"로 보정하는 구조적 접근
- 축적된 소스/자산 DB가 매 대화에 자동으로 관련 컨텍스트를 주입
- 사용자가 매번 배경 설명을 반복할 필요 없음

**설계 원칙**: 사용자가 컨텍스트를 구성하는 것이 아니라, 시스템이 컨텍스트를 구성해준다.

### 3.2 방법론 자동적용 (Auto Methodology) — 2순위

> **프롬프팅 스킬이 아니라, 도메인 전문성의 코드화.**

- "프롬프트를 잘 쓰는 법"을 가르치는 것이 아님
- 보고서 기획 방법론(구조화, 근거 배치, 요약 전략)이 하네스에 내장됨
- 하네스 엔지니어링 기반 품질 상향 평준화: 어떤 LLM을 써도, 누가 써도 일정 수준 이상의 산출물 보장
- 표준 산출물 포맷이 회사 양식을 자동 준수

**설계 원칙**: 사용자는 "무엇을 만들고 싶은지"만 말한다. "어떻게 만드는지"는 시스템이 안다.

### 3.3 점진적 자산화 (Progressive Asset Building) — 3순위

> **쓸수록 더 나아지는 환경. 그리고 조직의 데이터 레이크.**

#### 사용자 관점 (HR 담당자)
- 매 작업의 소스·초안·최종본이 개인 DB에 축적
- 다음 보고서 작성 시, 과거 자료가 자동 참조 → 점점 더 빠르고 정확한 결과물
- "3년치 보고서 맥락이 여기 있으니 다른 도구로 안 옮긴다" → 전환비용(Switching Cost) 형성

#### 운영 관점 (AAX CI Lab)
- 개인 DB들이 모여 HR 도메인 데이터 레이크 형성
- 핵심 자료의 아카이브화 → 조직 지식 자산으로 승격
- 인사이트 추출, 트렌드 분석 등 2차 활용 가능

**설계 원칙**: 개인의 작업이 곧 조직의 자산이 된다. 단, 개인 통제권은 유지한다.

## 4. 보안 포지셔닝: 제약이 아닌 선점 기회

### 기존 인식 (수동적)
> "사외 LLM을 못 쓰니까 어쩔 수 없이 사내 도구를 쓴다."

### 재정의된 포지셔닝 (능동적)
> "보안 환경에서도 사외 LLM 수준의 결과물을 낸다. 그리고 그 과정에서 축적된 자산은 사외 LLM이 열려도 대체 불가하다."

| 시점 | HARP의 가치 |
|------|------------|
| **사외 LLM 차단 중** (현재) | 유일한 AI 보고서 도구. 컨텍스트 설계로 사내 LLM 한계 보정 |
| **사외 LLM 개방 후** (미래) | 축적된 DB·컨텍스트·방법론이 전환비용. 사외 LLM을 백엔드로 연결하면 성능만 올라감 |

**핵심 논리**: 사외 LLM이 열리기 **전에** 컨텍스트 DB를 선점하는 것이 전략적 해자(moat)다.

## 5. 대상 사용자 프로필

### 주 사용자: HR 담당자 (People팀)

| 특성 | 설명 |
|------|------|
| AI 숙련도 | 낮음. ChatGPT 수준의 단순 질의 경험만 있음 |
| 핵심 니즈 | 반복되는 보고서를 빠르고 정확하게 만들고 싶음 |
| 가장 큰 장벽 | "뭘 어떻게 요청해야 할지 모르겠다" |
| 기대 경험 | "교육 결과 보고서 써줘" → 회사 양식의 초안이 나옴 |

### 운영자: AAX CI Lab

| 특성 | 설명 |
|------|------|
| 핵심 니즈 | HR 도메인 데이터의 체계적 축적·관리 |
| 기대 가치 | 개인 작업물 → 팀 자산 → 조직 데이터 레이크 점진적 전환 |

## 6. MVP 가치 범위

### 포함 (MVP)

| 가치 | 구현 형태 |
|------|----------|
| 컨텍스트 자동 주입 | 개인 DB 기반 관련 소스 자동 검색·첨부 |
| 방법론 자동적용 | 3대 산출물 유형별 하네스 (주간 HR 현황, 교육 운영 결과, 제도 검토 초안) |
| 표준 포맷 보장 | 회사 양식 템플릿 자동 적용 |
| 개인 자산 축적 | 소스 → 초안 → 최종본 라이프사이클 관리 |
| 선택적 공유 | 완성된 작업물의 공유 링크 생성 + 열람자 지정 |

### 로드맵 (v1.1+)

| 가치 | 구현 형태 |
|------|----------|
| 팀 공용 자산 DB | People팀 공유 소스 풀 + 검색 |
| 조직 데이터 레이크 | 승격된 자산의 중앙 아카이브 + 분석 |
| 산출물 유형 확장 | PPT, Excel 등 다양한 포맷 지원 |
| 사외 LLM 백엔드 전환 | 보안 정책 변경 시 LLM 교체 (하네스 구조 덕에 무중단) |

## 7. 경쟁 우위 요약

| vs. | HARP의 우위 |
|-----|------------|
| **사외 LLM (ChatGPT 등)** | 사내 데이터 활용 가능. 축적된 컨텍스트. 회사 양식 자동 적용 |
| **범용 사내 AI 챗봇** | HR 도메인 특화 방법론. 보고서 산출물 중심 설계. 자산 축적 구조 |
| **수작업 보고서 작성** | 초안 생성 시간 단축. 과거 자료 자동 참조. 품질 상향 평준화 |

## 8. 이 문서의 활용 가이드

### 설계 LLM에게

- 기능 우선순위 판단 시 **가치 삼각형 순서**(컨텍스트 > 방법론 > 자산화)를 따를 것
- "이 기능이 이중 간극 중 어느 쪽을 메우는가?"를 항상 확인할 것
- MVP 범위를 벗어나는 기능은 로드맵으로 분류할 것

### 구현 LLM에게

- 하네스 엔지니어링(CLAUDE.md)이 "방법론 자동적용"의 기술적 구현체임을 인지할 것
- 컨텍스트 주입 로직이 핵심 경쟁력이므로, 검색·매칭 품질에 집중할 것
- 개인 DB → 공유 → 조직 자산 흐름을 고려한 데이터 모델 설계할 것

### 제품 의사결정 시

- "이 기능이 세 가치 축 중 하나를 강화하는가?" → 아니면 후순위
- "이 기능이 전환비용을 높이는가?" → 그렇다면 우선순위 상향
- "초보 사용자가 추가 학습 없이 쓸 수 있는가?" → 아니면 UX 재설계
