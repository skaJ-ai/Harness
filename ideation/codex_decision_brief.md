# Codex Decision Brief

작성일: 2026-03-26
목적: 다른 LLM 또는 팀원에게 빠르게 전달하기 위한 현재 시점의 정제된 의사결정 요약
기준: 아직 최종 확정 전이며, 현재까지 대화에서 도출된 `가장 강한 결론`만 남김

## 1. 제품 한 줄 정의

> HR 담당자가 자기 작업 방식으로 일하면서, AI와 대화한 내용을 회사 표준 산출물로 빠르게 정리하고, 그 과정에서 재사용 가능한 HR 자산을 점진적으로 쌓는 private-first 작업공간

## 2. 핵심 철학

- 이 제품은 `LLM을 믿는 제품`이 아니라 `컨텍스트를 설계하는 제품`이어야 한다
- 대화는 자산이 아니라 원재료다
- 자산은 정제되거나 승격된 결과물이어야 한다
- 모델이 기억하는 것이 아니라, 플랫폼이 사용자 맥락을 저장하고 필요할 때 다시 공급해야 한다

## 3. 누구를 위한 제품인가

- 범용 직군 대상 아님
- `HR 도메인 AX`에 집중
- 주요 사용자는 소수~수백 명 규모의 HR 담당자
- 초기 목표는 전사 시스템 통합이 아니라 `실제 HR 실무자가 반복적으로 쓰는 작업공간`을 만드는 것

## 4. 이 제품이 해결하려는 핵심 문제

1. HR 자료가 Word, PPT, Excel, 메모, 대화 등에 흩어져 있다
2. 매번 같은 배경설명을 반복해야 한다
3. 결과물 형식과 품질이 사람마다 흔들린다
4. 좋은 초안과 근거자료가 다음 작업에 잘 재사용되지 않는다
5. LLM이 바뀌면 작업 맥락이 끊긴다

## 5. 왜 개인 작업공간이 핵심인가

개인 작업공간은 보조 기능이 아니라 제품 핵심이다.

이유:

- LLM은 안정적인 장기 기억을 갖지 않으므로, 시스템이 사용자 맥락을 대신 기억해야 한다
- 로컬 폴더에만 쌓으면 문서-근거-초안-최종본 연결이 어렵다
- 과거 작업을 다시 시작할 때 맥락 복원이 어렵다
- 모델이 바뀌어도 작업 맥락이 유지돼야 한다
- HR 업무는 기본적으로 private-first가 맞다

정확한 표현:

- 개인 작업공간이 `외부 LLM의 장기 기억을 활성화`하는 것이 아니다
- 더 정확히는 `개인 작업공간이 LLM의 기억을 대신한다`

## 6. 현재 MVP 의사결정

### 포함

- HR 담당자 1인용 `private workspace`
- LLM과의 대화 기반 ideation
- 텍스트 / 복사-붙여넣기 기반 입력
- 표준 산출물 3종 우선 지원
- 명시적 `정리하기` 트리거로 산출물 생성
- 세션 일부를 자산으로 승격 저장
- workspace 내부 자료를 활용한 retrieval

### 제외 또는 후순위

- 범용 직군 확장
- Team shared / Org template full workflow
- 대규모 시스템 연동
- Knowledge Graph 시각화
- Office import pipeline
- 화려한 PPT/Excel 자동화
- 과도한 멀티에이전트 구조

## 7. MVP 표준 산출물 3종

현재 가장 현실적인 후보 3종을 아래처럼 고정한다.

1. `주간 HR 현황 보고`
- 반복성이 높음
- 표준 형식 정의가 쉬움
- 빠른 초안 생성 가치가 분명함

2. `교육 운영 결과 요약`
- 입력 자료가 비교적 구조화되어 있음
- 결과 요약/인사이트 정리가 쉬움
- 근거자료 연결이 가능함

3. `제도 검토 초안`
- HR 실무 가치가 큼
- 근거자료, 비교안, 리스크 정리 등 템플릿화 가능
- 다만 최종본은 사람 검토가 강하게 필요

현재 제외:

- `면담 결과 정리`
이유: 민감도와 보안 이슈가 더 커서 파일럿 초기엔 부담이 큼

## 8. 사용자 여정 1개

예시 시나리오: `교육 운영 결과 요약 작성`

1. 사용자가 private workspace에서 새 작업을 시작한다
2. 교육 결과 관련 메모/표/텍스트를 복사-붙여넣기 한다
3. AI와 대화하며 핵심 이슈와 포인트를 정리한다
4. 사용자가 `정리하기`를 누른다
5. 시스템이 `교육 운영 결과 요약` 템플릿에 맞춰 draft를 생성한다
6. 사용자가 draft를 수정/보완한다
7. 결과물을 final_deliverable로 저장한다
8. 재사용 가치가 높으면 사용자가 promoted_asset으로 승격한다

핵심:

- 산출물 전환은 자동이 아니라 `명시적 트리거`
- 모든 세션이 자산이 되는 것이 아니라, `선별 승격` 구조

## 9. 데이터 라이프사이클

저장 단위는 아래처럼 구분한다.

- `session`: 사용자와 AI의 작업 세션
- `source`: 붙여넣은 근거자료, 참고 텍스트, 메모
- `draft`: 템플릿에 맞춰 생성된 초안
- `final_deliverable`: 사용자가 확정한 결과물
- `promoted_asset`: 다시 쓰기 위해 승격한 자산

핵심 원칙:

- 작업은 저장한다
- 자산은 승격한다
- 모든 것을 자동 자산화하지 않는다

최소 메타데이터:

- 문서 유형
- 주제
- 작성자
- 생성일
- 버전
- 공개 범위
- 근거 출처
- 관련 이전 문서
- 사용한 템플릿

## 10. 계정과 권한

초기 파일럿 기준:

- self-signup 허용
- 입력값: `loginId`, `password`, `name`, `employeeNumber`, `knoxId`
- 가입 즉시 개인 workspace 생성
- 기본 권한은 본인 private workspace만 접근 가능
- 관리자 계정은 사용자 목록, 계정 상태, 내역 확인 가능

운영상 최소 필요사항:

- `loginId`, `employeeNumber`, `knoxId` 유니크 제약
- 비밀번호 해시 저장
- 기본값 `private`
- 공유/권한변경/계정상태 변경 로그

현재 판단:

- 파일럿 단계에서는 강한 인증보다 `작업공간 분리`와 `인가 통제`가 더 중요하다

## 11. Retrieval MVP

현재 단계에서는 full RAG를 전제로 가지 않는다.

MVP retrieval 원칙:

- 같은 workspace 내부 자료만 대상으로 함
- 문서 유형 + 최근성 + 키워드 기준으로 후보를 좁힘
- 상위 소수의 source만 evidence bundle로 묶음
- 긴 문서는 전체 투입 대신 섹션/요약 단위로 사용
- 생성은 가능하면 section-by-section 방식으로 수행

즉:

- 지금 필요한 건 `vector DB 대공사`가 아니라
- `작고 정확한 evidence bundle`을 만드는 정책이다

## 12. 하네스 개념을 제품 기능으로 번역

이 제품이 단순 AI 채팅툴과 다른 이유는 `하네스`를 제품 기능으로 녹이기 때문이다.

### Product Front Guard

- 산출물별 템플릿 강제
- 필수 입력 항목 강제
- 회사 표준 섹션 구조 강제
- 형식/톤/문서 타입 선택 강제
- 근거 없이 단정적인 서술 제한

### Product Back Guard

- 출처 포함 여부 점검
- 내부 규정 충돌 여부 점검
- 민감정보 노출 여부 점검
- 완성도 / 신뢰도 / 재사용성 평가
- 필요 시 `추정`, `불확실`, `근거 부족` 표시

핵심:

- 개발 프로세스에서만 하네스를 쓰는 것이 아니라
- 사용자 산출물 생성 단계에서도 Guard 개념이 적용되어야 한다

## 13. DB 축적의 현실적 가치

냉정한 판단:

- DB가 쌓인다고 자동으로 더 좋은 기획/보고서가 나오지는 않는다
- 하지만 `잘 구조화된 DB`는 분명히 유의미한 차이를 만든다

DB가 유의미한 이유:

- 비슷한 안건의 과거 산출물을 재사용 가능
- 조직이 선호하는 표현/형식/논리를 일관되게 유지 가능
- 이전 의사결정과 근거자료를 다시 찾을 수 있음
- LLM이 무에서 생성하는 것이 아니라 조직 맥락 위에서 생성하게 됨

DB가 무의미해지는 조건:

- 대화 로그만 무차별 저장
- 초안/최종본/근거자료 구분 없음
- 메타데이터 없음
- 검색과 재활용이 어려움

결론:

- 좋은 DB는 `전자 창고`가 아니라 `맥락 복원 장치`여야 한다

## 14. Knowledge Graph에 대한 현재 판단

가능은 하지만 MVP 핵심은 아니다.

우선순위:

1. 문서/자산 저장
2. 메타데이터와 링크 저장
3. 검색과 재호출
4. 필요해지면 그래프화

그래프보다 먼저 중요한 것:

- 검색
- 출처 추적
- 관련 문서 연결

## 15. 보고서 딸깎 가능성

현실적인 표현은 아래와 같다.

- `초안 딸깎`은 가능
- `최종본 딸깎`은 문서 유형에 따라 제한적

이 제품은 `완성 보고서 버튼`보다 아래 가치를 약속하는 것이 맞다.

- 표준 형식의 초안을 매우 빠르게 만든다
- 근거자료를 다시 활용하게 해준다
- 결과물 형식을 덜 흔들리게 만든다
- 좋은 작업이 다음 문서의 기반이 되게 한다

## 16. 비개발자 대상 메시지

비개발자에게는 기술 용어보다 업무 효용으로 설명한다.

권장 한 줄 설명:

> HR 담당자가 흩어진 자료를 바탕으로 AI와 대화하면, 회사 표준 형식의 결과물을 더 빠르고 일관되게 만들고, 그 과정이 나중에 다시 쓸 수 있는 자산으로 쌓이는 작업공간

## 17. 아직 남은 핵심 질문

1. 각 표준 산출물의 실제 템플릿 초안은 어떻게 생길 것인가
2. `promoted_asset` 승격 UX는 어떻게 할 것인가
3. 기존 `HR-Process-Coaching-AI`와의 관계를 어떻게 정의할 것인가
4. 장기적으로 shared workspace를 언제 어떤 방식으로 열 것인가
