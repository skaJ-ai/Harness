# Codex Ideation Notes

작성일: 2026-03-26

## 1. 현재 이해한 제품 정의

이 프로젝트는 범용 문서 생성기가 아니라, HR 도메인 전용 AX 플랫폼에 가깝다.

핵심 목적은 다음과 같다.

- 어떤 LLM을 써도 업무 산출물의 품질 하한선을 일정하게 유지한다.
- HR 담당자가 대화 중심으로 일하되, 결과는 표준 산출물 형식으로 정리되게 한다.
- 작업 과정에서 생성된 초안, 대화, 근거자료, 결과물을 점진적으로 자산화한다.
- 나중에 사내 시스템과 연결될 수 있도록 작업 맥락과 산출물 구조를 일관되게 남긴다.

한 줄 정의:

> HR 담당자가 자기 작업 방식으로 일하면서, 대화와 초안과 근거자료를 표준 산출물로 정리하고, 그 과정에서 재사용 가능한 HR 자산 DB를 점진적으로 쌓는 하네스 플랫폼

## 2. 합의된 방향

### 맞는 방향

- 범용 직군 대응이 아니라 HR 도메인에 집중한다.
- 처음부터 모든 데이터를 밀어넣지 않는다.
- "쓰면 쓸수록 DB화된다"가 더 정확한 표현이다.
- 보고서를 사람이 처음부터 끝까지 수동 작성하는 방식보다, LLM과 대화하다가 표준 산출물로 정리되는 흐름이 맞다.
- 다만 완전 자유 대화가 아니라 표준 보고서/모판/템플릿 제약이 필요하다.

### 중요한 해석

- 이 플랫폼의 핵심은 "문서 생성"보다 "작업공간 + 표준화 + 점진적 자산화"이다.
- 대화 자체는 자산이 아니라 원재료다.
- 자산은 정제되거나 공유 승격된 결과물이어야 한다.

## 3. 하네스 관점에서 본 장점

- 프롬프트 품질에 기대지 않고 규칙과 절차로 LLM 출력을 통제하려는 방향이 맞다.
- `Front Guard + Back Guard` 구조는 팀 운영 관점에서 확장 가능성이 있다.
- Markdown 중심 흐름은 사람, AI, 후처리 파이프라인 모두에 유리하다.
- 인터뷰어/기획자/작성자/리뷰어처럼 역할 분리를 두는 사고 자체는 타당하다.
- `PROJECT-CONTEXT.md` 같은 핸드오프 문서는 실제 협업에 유용하다.

## 4. 현재 보완이 필요한 점

### 하네스 실행 가능성

- 현재 `npm run harness:check`는 self-hosting 상태가 아니다.
- `@typescript-eslint/naming-convention` typed lint 설정이 불완전하다.
- `CLAUDE.md`는 `CHECKLIST.md`를 요구하지만 파일이 없다.
- 문서에 적힌 규칙과 실제 실행 범위가 일부 어긋난다.

### 제품 정의 명확화

- 기존 `README.md`는 "범용 하네스 템플릿" 톤이 강하다.
- 실제로는 HR 도메인 전용 AX 플랫폼 쪽으로 정의가 이동하고 있다.
- 따라서 README/컨텍스트 문서를 현재 정체성에 맞게 갱신할 필요가 있다.

### 범위 관리

- PPT/엑셀/그래프/대규모 연동까지 초기에 다 가져가면 제품이 무거워진다.
- MVP는 "개인 작업 -> 표준 산출물 생성 -> 선택적 공유 -> 점진적 DB화"에 집중하는 편이 낫다.

## 5. 개인 작업공간에 대한 판단

개인 작업공간은 옵션이 아니라 핵심 구조에 가깝다.

이유:

- HR 업무는 초안과 민감정보가 많아 기본값이 공유면 안 된다.
- 사용자는 자신이 익숙한 흐름으로 먼저 일하고 싶어한다.
- 실제 자산화는 개인 작업물이 곧바로 공유되는 방식보다, 검토 후 승격되는 방식이 더 안전하다.

권장 구조:

- `Private Workspace`: 개인 작업, 대화, 초안, 메모, 임시 자료
- `Team Shared Space`: 팀 내 공유 문서, 리뷰 가능한 결과물
- `Org Template Space`: 표준 보고서 양식, 승인된 모판, 공통 템플릿

기본 원칙:

- 기본값은 `private`
- 공유는 명시적 승격이어야 함
- 모든 산출물에는 소유자, 공개 범위, 버전, 출처, 민감도 메타데이터가 필요함

## 6. 계정/식별자에 대한 현재 판단

질문:

> 공식 시스템 등록 전에도 계정을 강제로 만들고, 사번/Knox ID 같은 식별자를 미리 받는 것이 맞는가?

현재 판단:

- "개인 작업공간"을 운영하려면 최소한의 사용자 식별자는 초기에 필요하다.
- 다만 현재 논의 기준에서는 사용자 풀이 HR 담당자 수백 명 규모이고, 악의적 오가입 가능성을 높게 보지 않는다.
- 따라서 초기 파일럿 단계에서는 복잡한 SSO보다 `간단한 가입 + 개인 작업공간 생성`이 더 현실적이다.

현재 합의에 가까운 접근:

1. 초기 파일럿 / MVP
- self-signup 허용
- 입력값은 최소 `loginId`, `password`, `name`, `employeeNumber`, `knoxId`
- 가입 즉시 개인 workspace 생성
- 기본 권한은 본인 private workspace만 접근 가능

2. 관리자 기능
- 관리자 계정은 사용자 목록, 기본 내역, 계정 상태를 확인 가능
- 사용자 비활성화, 권한 변경, 공유 권한 통제 가능
- 초기에는 관리자 계정 1개로 시작할 수 있으나, 장기적으로는 admin 역할 분리를 고려

3. 나중 단계
- 공식 시스템 연동이 필요해지면 사번/Knox/SSO 체계와 매핑
- 기존 사용자/작업물과 정식 계정 체계를 연결 가능한 구조로 유지

핵심 판단:

- `워크스페이스 분리`는 초기에 필요하다.
- 초기에는 `강한 인증`보다 `작업공간 분리`와 `인가 범위 통제`가 더 중요하다.
- 즉, 지금 중요한 것은 "누가 들어오느냐"보다 "들어온 뒤 어디까지 보이고 공유할 수 있느냐"이다.

운영상 최소 필요사항:

- `loginId`, `employeeNumber`, `knoxId` 유니크 제약
- 비밀번호 해시 저장
- 기본값 `private`
- 공유/권한변경/계정상태 변경 로그

주의:

- 단순 입력만으로 사번/Knox ID 진위를 완전히 검증하는 구조는 아니다.
- 다만 현재 파일럿 전제에서는 이 리스크보다 사용자 마찰을 줄이는 쪽이 더 중요하다는 판단이다.

이 제품의 핵심은 "글을 잘 써주는가"가 아니다.

그리고 그 결과가 나중에 재사용 가능한 자산으로 남는가.

## 10. 담당자 계정/작업공간에 대한 추가 정리

현재 제품 관점에서 더 중요한 것은 `인증`보다 `인가`다.

- 회원가입은 가볍게 갈 수 있다.
- 하지만 모든 사용자가 기본적으로 공용 자산을 보거나 수정하게 하면 안 된다.
- 가입 직후에는 개인 private workspace만 제공하는 것이 맞다.
- 팀 공유 공간, 공통 템플릿 공간, 자산 승격 권한은 별도 통제가 필요하다.

즉, 설계 우선순위는 다음이다.

1. 사용자별 private workspace
2. 공유 범위 제어
3. 자산 승격 흐름
4. 관리자 가시성
5. 정식 사내 인증 연동

## 11. Office 문서 입력 제약과 가져오기 전략

### 현재 전제

- HR 담당자가 실제로 LLM과 아이디에이션할 때 넣고 싶은 입력은 Word, PPT, Excel인 경우가 많다.
- 단순 브라우저 업로드는 사내 보안정책(Nasca) 때문에 막힐 수 있다.
- 따라서 "웹 업로드"를 전제로 한 입력 설계는 현실성이 낮을 수 있다.

### 중요한 판단

- 이 문제의 본질은 "파일 파서"가 아니라 "입력 경로"다.
- 브라우저 업로드가 막히면, 서버 업로드를 우회하는 꼼수보다 `로컬 측 파싱 -> 구조화된 내용 전달` 쪽이 더 현실적이다.
- 즉, 이 플랫폼은 웹에서 raw file을 직접 받는 방식보다, 로컬 helper 또는 Office 연동 모듈을 통해 내용을 추출하는 구조가 더 맞다.

### xlwings에 대한 판단

- `xlwings`는 Excel 쪽에는 유효하다.
- 하지만 Word와 PPT까지 일반적으로 해결하는 범용 Office 업로더로 보면 안 된다.
- Excel 입력 경로에 대해서는 다음 두 가지가 가능하다.

1. 로컬 Excel 연동형
- 사용자의 PC에서 Excel이 열려 있는 상태에서 `xlwings`로 시트/범위/테이블을 읽어 구조화
- 장점: 사용자가 보고 있는 실제 시트 기준 추출 가능
- 단점: 로컬 Python/Excel 환경 의존성이 있음

2. 저장된 파일 읽기형
- `xlwings Reader`는 읽기 전용 Excel 파일 리더를 제공
- `xlsx`, `xlsm`, `xlam`, `xlsb`, `xls` 포맷을 읽을 수 있음
- 다만 xlwings PRO 기능이며, 읽기 전용이고 Excel 전용이다

### Word / PPT에 대한 판단

- Word는 `python-docx` 같은 방식으로 `.docx`를 읽어 텍스트/표 구조를 추출하는 편이 현실적이다
- PPT는 `python-pptx`로 `.pptx`의 슬라이드, 텍스트, 표 등을 읽거나 갱신할 수 있다
- 즉, 입력 파이프라인은 `xlwings 하나로 통일`이 아니라 `파일 유형별 어댑터`가 맞다

### 더 현실적인 제품 구조

- 브라우저 업로드 대신 `Import from Office` 계층을 둔다
- 각 입력 방식은 아래처럼 분기한다
  - Excel: `xlwings` 또는 별도 Excel parser
  - Word: `python-docx` 또는 Office COM/로컬 변환
  - PPT: `python-pptx` 또는 Office COM/로컬 변환

권장 흐름:

1. 사용자가 로컬 Office 문서에서 "가져오기" 실행
2. 로컬 helper가 파일을 읽어 텍스트/표/슬라이드 구조를 추출
3. 추출 결과를 Markdown 또는 JSON으로 정규화
4. 플랫폼에는 정규화된 내용만 전달
5. 필요 시 원본 파일 경로나 메타데이터만 참조로 저장

### 핵심 결론

- "단순 파일 업로드"가 막히는 환경이라면, 업로드 우회보다 `로컬 측 추출 -> 구조화 전달` 아키텍처가 맞다
- `xlwings`는 Excel 입력 어댑터로는 유효하지만 Word/PPT까지 해결하지는 못한다
- 따라서 이 기능은 `업로드 기능`이 아니라 `로컬 Office import pipeline`으로 정의하는 편이 정확하다

### 외부 기술 참고

- xlwings: https://docs.xlwings.org/en/stable/
- xlwings Reader: https://docs.xlwings.org/en/stable/pro/reader.html
- xlwings with other Office Apps: https://docs.xlwings.org/en/stable/other_office_apps.html
- python-docx: https://python-docx.readthedocs.io/en/latest/
- python-pptx: https://python-pptx.readthedocs.io/en/latest/

## 12. 다른 LLM 피드백 요약

다른 LLM이 준 피드백 중 유효한 핵심은 다음과 같다.

### 강하게 동의한 포인트

- `대화는 원재료, 자산은 정제된 결과물`이라는 구분은 중요하다
- 파일럿 단계에서는 인증보다 workspace isolation이 더 중요하다
- HR 도메인에서는 default `private`, 공유는 명시적 승격이 맞다
- 범용 확장, 그래프, 화려한 산출물 자동화는 MVP에서 미루는 편이 맞다

### 추가로 보완해야 한다고 본 포인트

- `표준 산출물`이 구체적으로 정의되지 않았다
- `쓰면 쓸수록 DB화된다`는 비전은 있으나, 어떤 단위가 저장되고 승격되는지는 아직 불명확하다
- `대화 -> 산출물 전환 시점`이 자동인지 수동인지 정해야 한다
- 기존 `HR-Process-Coaching-AI`와 HARP의 관계를 명확히 해야 한다

### 받아들인 조정

- MVP에서 `Workspace 3-tier`는 과하다
- 우선은 `Private workspace only`가 현실적이다
- Office import / local helper는 장기 방향으로 남기고, MVP 핵심에서는 뒤로 미루는 편이 맞다

## 13. 비개발자 대상 제품 설명 방식

비개발자에게는 `하네스`, `멱등성`, `Markdown`, `Knowledge Graph` 같은 용어를 앞에 내세우면 전달이 어렵다.

설명은 다음 순서가 더 적절하다.

1. 지금 HR 담당자가 겪는 문제
- 자료가 Word, PPT, Excel, 메모 등에 흩어져 있다
- 매번 새로 정리하고 새로 초안을 만든다
- 사람마다 결과물 형식과 품질이 달라진다
- 좋은 작업물이 다음 업무에 잘 재사용되지 않는다

2. 이 툴이 하는 일
- 담당자가 자료를 바탕으로 AI와 대화한다
- 그 대화를 회사 표준 형식의 산출물로 정리한다
- 작업 과정은 개인 작업공간에 남는다
- 좋은 결과물은 나중에 다시 꺼내 쓸 수 있는 자산으로 남는다

3. 사용자가 얻는 효용
- 더 빨리 초안을 만든다
- 결과물 형식이 덜 흔들린다
- 과거 작업을 재사용하기 쉬워진다
- 특정 LLM이 바뀌어도 업무 방식이 덜 흔들린다

비개발자용 한 줄 설명:

> HR 담당자가 흩어진 자료를 바탕으로 AI와 대화하면, 회사 표준 형식의 결과물을 더 빠르고 일관되게 만들고, 그 과정이 나중에 다시 쓸 수 있는 자산으로 쌓이는 작업공간

## 14. 개인 작업공간이 왜 핵심인가

개인 작업공간은 보조 기능이 아니라, 이 제품이 `조금 더 나은 채팅창`을 넘어 `업무 시스템`이 되기 위한 핵심 구조일 가능성이 높다.

이유:

- LLM 자체는 안정적인 장기 기억을 갖지 않으므로, 시스템이 사용자 맥락을 대신 기억해야 한다
- 로컬 폴더에 흩어진 자료는 문서-근거-초안-최종본의 연결이 어렵다
- 같은 작업을 다시 시작할 때 과거 배경설명과 초안이 쉽게 복원되지 않는다
- 모델이 바뀌어도 사용자 맥락이 플랫폼에 남아야 LLM 독립성이 생긴다
- HR 업무는 private 초안과 공유 가능한 결과물을 분리해야 한다

정확한 표현:

- 개인 작업공간이 `외부 LLM의 장기 기억을 활성화`하는 것은 아님
- 더 정확히는 `개인 작업공간이 LLM의 기억을 대신한다`
- 즉, 모델이 기억하는 것이 아니라 플랫폼이 사용자 맥락을 저장하고, 필요한 순간 retrieval로 다시 공급하는 구조다

개인 작업공간이 없으면 생기는 문제:

- 매번 같은 배경설명을 반복해야 함
- 예전 작업물 재사용이 어려움
- 보고서와 근거자료의 연결이 끊김
- LLM이 바뀌면 맥락이 초기화됨
- 개인 초안과 조직 자산 사이 경계가 흐려짐

## 15. DB가 쌓이면 진짜 더 나아지는가

냉정하게 말하면, `DB가 쌓인다고 자동으로 더 좋은 기획/보고서가 나오지는 않는다`.

하지만 `잘 구조화된 DB`는 분명히 유의미한 차이를 만든다.

의미가 생기는 이유:

- 비슷한 HR 안건의 과거 산출물을 재사용할 수 있다
- 조직이 선호하는 표현, 논리, 형식을 일관되게 유지할 수 있다
- 이전 의사결정, 근거자료, 참고 사례를 다시 찾을 수 있다
- LLM이 빈칸에서 쓰는 것이 아니라 조직 맥락을 바탕으로 쓰게 된다

의미가 사라지는 경우:

- 대화 로그만 무차별 저장
- 초안, 최종본, 근거자료 구분 없음
- 작성자, 버전, 출처, 문서 유형 메타데이터 없음
- 검색해도 재사용 가치가 낮은 자료만 누적

즉, 좋은 DB는 `전자 창고`가 아니라 `맥락 복원 장치`여야 한다.

## 16. 점진적 DB화의 최소 단위

`쓰면 쓸수록 DB가 된다`를 실제 제품으로 바꾸려면 저장 단위를 나눠야 한다.

최소 구분 단위:

- `session`
- `source`
- `draft`
- `final_deliverable`
- `promoted_asset`

최소 메타데이터:

- 문서 유형
- 주제
- 작성자
- 생성일
- 버전
- 공개 범위
- 근거 출처
- 관련 이전 문서
- 사용한 템플릿

핵심은 `모든 것을 자동 자산화`하는 것이 아니라, `작업은 저장하고 자산은 승격`하는 구조다.

## 17. Knowledge Graph에 대한 현실적 판단

그래프화는 가능하다. 다만 `가능`과 `초기 핵심 가치`는 다르다.

그래프가 의미 있으려면 관계가 구조화되어 있어야 한다.

예시 관계:

- 문서 A `derived_from` 문서 B
- 문서 A `references` 규정 문서 B
- 문서 A `about` 특정 제도/팀/지표
- 템플릿 A가 문서 B/C/D의 기반

따라서 올바른 우선순위는 다음이다.

1. 문서/자산 저장
2. 메타데이터와 링크 저장
3. 검색과 재호출
4. 필요해지면 그래프화

초기에는 `Knowledge Graph`보다 `검색`, `출처 추적`, `관련 문서 연결`이 더 중요하다.

## 18. 보고서 딸깎 가능성에 대한 냉정한 판단

`딸깎 최종본`을 전면 약속하면 위험하다.

더 정확한 표현:

- `초안 딸깎`은 가능
- `최종본 딸깎`은 문서 유형에 따라 제한적

자동화 가능성이 높은 경우:

- 주간 HR 현황 보고
- 교육 운영 결과 요약
- 면담 결과 정리
- 제도 검토 초안

사람 검토가 강하게 필요한 경우:

- 민감한 평가제도 개편안
- 조직개편 관련 보고
- 임원 설득용 정무 문서

따라서 제품 메시지는 `버튼 한 번에 완성 보고서`보다 아래 표현이 더 현실적이다.

- 표준 형식의 초안을 매우 빠르게 만든다
- 근거자료를 다시 활용하게 해준다
- 결과물 형식을 덜 흔들리게 만든다
- 좋은 작업이 다음 문서의 기반이 되게 한다

## 19. 5대 핵심 고민에 대한 현재 판단

### 1. 데이터 사일로

부분적으로 해결 가능하다.

- 모든 시스템을 한 번에 통합하는 방식보다
- 사용 과정에서 붙여넣은 근거, 생성된 산출물, 메모, 초안을 같은 작업공간 안에서 연결하는 방식이 현실적이다

### 2. AI 답변 신뢰성

구조를 잘 잡으면 개선 가능하다.

필요 조건:

- 내부 규정/근거 문서 retrieval 우선
- 답변과 산출물에 출처 연결
- 근거가 없으면 추정/불확실로 표시
- 표준 템플릿으로 출력 형식 강제

### 3. HR 민감 데이터

기술만으로 해결되지 않는다. 운영 설계가 중요하다.

필요 조건:

- private workspace 기본
- 공개 범위 분리
- 계정/권한/로그 관리
- 외부 LLM으로 무엇을 보내는지 통제
- 민감정보 마스킹 또는 제한

### 4. AI 활용 신규 경험

오히려 강점이 될 수 있다.

단순 채팅을 넘어:

- 개인 작업공간
- 이전 초안/근거 재사용
- 표준 산출물 전환
- 승격 저장

이 흐름을 제공하면 `업무 인터페이스`로서 차별화된다.

### 5. 많은 데이터를 핵심만 추려 컨텍스트로 넣을 수 있는가

이 부분이 매우 중요하다.

- DB 전체를 넣는 방식은 실패 확률이 높다
- 필요한 것은 retrieval와 compression이다

필요 전략:

- workspace 범위 제한
- 문서 유형 필터
- 최근성/관련성 점수화
- 근거 묶음 추출
- 긴 문서는 섹션별 요약 후 사용
- 보고서는 section-by-section 생성

핵심 결론:

- 이 제품은 `LLM을 믿는 제품`이 아니라 `컨텍스트를 설계하는 제품`이어야 한다

## 20. 모델 컨텍스트 한계에 대한 현재 판단

Qwen 계열이나 사내 모델이 긴 컨텍스트를 지원하더라도, 제품 설계는 `큰 컨텍스트를 전제로` 가면 안 된다.

핵심 원칙:

- 모델의 context window가 크더라도, 전체 DB를 한 번에 넣지 않는다
- retrieval와 compression을 통해 지금 필요한 핵심만 넣는다
- raw session보다 structured note와 evidence bundle을 우선 사용한다

현실적 아키텍처:

1. 원본 대화/문서 저장
2. 중간 구조화 노트 생성
3. 관련 근거 문서 retrieval
4. evidence bundle 압축
5. 산출물은 section-by-section 생성
6. 마지막에 review/rewrite 단계 수행

따라서 질문은 `Qwen 3.5나 GAUSS가 다 처리할 수 있느냐`보다 `우리가 필요한 컨텍스트를 작고 정확하게 만들 수 있느냐`가 더 중요하다.

## 21. 현재 시점의 MVP 재정의

현재까지 논의를 기준으로, MVP는 아래처럼 압축하는 것이 현실적이다.

- HR 담당자 1인용 private workspace
- LLM과 대화 기반 ideation
- 텍스트/복사-붙여넣기 기반 입력
- 표준 산출물 2~3종만 우선 지원
- 명시적 `정리하기` 트리거로 산출물 생성
- 세션 일부를 자산으로 승격 저장
- retrieval 기반 컨텍스트 공급

MVP에서 뒤로 미룰 가능성이 높은 것:

- Team shared / Org template full workflow
- 대규모 시스템 연동
- Knowledge Graph 시각화
- Office import pipeline
- 화려한 PPT/Excel 자동화
